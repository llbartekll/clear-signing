# react-native-clear-signing

ERC-7730 v2 clear-signing for React Native (iOS) â€” JSI/TurboModule bindings over the [`clear-signing`](https://crates.io/crates/clear-signing) Rust crate.

Generated by [`uniffi-bindgen-react-native`](https://github.com/jhugman/uniffi-bindgen-react-native) on top of the existing UniFFI surface in `crates/clear-signing/src/uniffi_compat/mod.rs`.

## Status

- iOS: ready (Hermes + New Architecture, RN 0.81+)
- Android: in progress

## Install

```sh
npm install react-native-clear-signing uniffi-bindgen-react-native
```

The `uniffi-bindgen-react-native` package is the JSI runtime â€” required at install time, not just dev. Don't pin it to a different major version than what `react-native-clear-signing` expects (currently `0.31.0-2`).

### iOS

Add this line to your app's `ios/Podfile`, inside the target block:

```ruby
pod 'uniffi-bindgen-react-native', :path => '../node_modules/uniffi-bindgen-react-native'
```

Then:

```sh
cd ios && bundle exec pod install
```

The `pod install` step downloads the prebuilt XCFramework from the matching GitHub release â€” no Rust toolchain needed on the consumer side.

### Requirements

- React Native 0.81+ (older RN versions have not been validated)
- New Architecture enabled (`RCT_NEW_ARCH_ENABLED=1`)
- Hermes (default in RN 0.81)
- iOS 13.4+

## Usage

`DataProviderFfi` callbacks are **synchronous** across the FFI boundary â€” they cannot
return Promises. The integration pattern is therefore: resolve descriptors (async), then
pre-load any token / proxy / ENS data into in-memory maps before calling the format step.

```ts
import {
  DescriptorResolutionOutcome_Tags,
  FormatOutcome_Tags,
  clearSigningFormatCalldata,
  clearSigningResolveDescriptorsForTx,
  type DataProviderFfi,
  type DescriptorResolutionOutcome,
  type FormatOutcome,
  type TokenMetaFfi,
  type TransactionInput,
} from 'react-native-clear-signing';

// Cache key is `${chainId}:${address.toLowerCase()}`.
class WalletDataProvider implements DataProviderFfi {
  constructor(
    private readonly tokens: Map<string, TokenMetaFfi> = new Map(),
    private readonly proxies: Map<string, string> = new Map(),
    private readonly ensNames: Map<string, string> = new Map(),
  ) {}

  resolveToken(chainId: bigint, address: string): TokenMetaFfi | undefined {
    return this.tokens.get(`${chainId}:${address.toLowerCase()}`);
  }
  resolveEnsName(address: string): string | undefined {
    return this.ensNames.get(address.toLowerCase());
  }
  resolveLocalName(): string | undefined { return undefined; }
  resolveNftCollectionName(): string | undefined { return undefined; }
  resolveBlockTimestamp(): bigint | undefined { return undefined; }
  getImplementationAddress(chainId: bigint, address: string): string | undefined {
    return this.proxies.get(`${chainId}:${address.toLowerCase()}`);
  }
}

async function clearSign(tx: TransactionInput) {
  // 1. Resolve descriptors (async â€” hits the GitHub registry, falls back to
  //    proxy lookup via DataProviderFfi.getImplementationAddress when needed).
  //    Pre-seed any proxy implementations the wallet already knows about so
  //    step 1 doesn't hit the wallet's RPC mid-resolve.
  const knownProxies = new Map([['1:0x1111111111111111111111111111111111111111', '0x6666â€¦']]);
  const resolveProvider = new WalletDataProvider(new Map(), knownProxies);
  const resolution: DescriptorResolutionOutcome =
    await clearSigningResolveDescriptorsForTx(tx, resolveProvider);
  if (resolution.tag === DescriptorResolutionOutcome_Tags.NotFound) {
    return null;
  }
  const descriptors = resolution.inner[0];

  // 2. Pre-load token metadata for any addresses the formatter is likely to
  //    query. Wallets typically inspect the calldata or descriptor here and
  //    fetch via their own async paths (RPC, indexer, local cache).
  const tokens = new Map<string, TokenMetaFfi>([
    ['1:0xdac17f958d2ee523a2206206994597c13d831ec7', { symbol: 'USDT', decimals: 6, name: 'Tether USD' }],
  ]);

  // 3. Format with a fresh provider holding the populated maps. The
  //    DataProviderFfi callbacks fire synchronously during the async Rust
  //    call, so they must read from already-populated state.
  const formatProvider = new WalletDataProvider(tokens, knownProxies);
  const outcome: FormatOutcome = await clearSigningFormatCalldata(
    descriptors,
    tx,
    formatProvider,
  );

  if (outcome.tag === FormatOutcome_Tags.ClearSigned) {
    const { model, diagnostics } = outcome.inner;
    return { intent: model.intent, entries: model.entries, diagnostics };
  } else {
    const { model, reason, diagnostics } = outcome.inner;
    return { intent: model.intent, entries: model.entries, fallbackReason: reason, diagnostics };
  }
}
```

See `crates/clear-signing/src/uniffi_compat/mod.rs` for the full FFI surface (`clearSigningFormatCalldata`, `clearSigningFormatTypedData`, `clearSigningResolveDescriptor`, `clearSigningResolveDescriptorsForTx`, `clearSigningMergeDescriptors`).

## Local development

Regenerate the package after editing the Rust crate:

```sh
./scripts/build-rn-ios.sh
```

This runs `ubrn build ios --sim-only --and-generate` and re-applies the podspec patches via `scripts/prepare-rn-podspec.sh`. The local XCFramework lands at `bindings/react-native/ClearSigning.xcframework/` (gitignored).

To smoke-test against a fresh RN app:

```sh
npx @react-native-community/cli init ClearSigningExample \
  --version 0.81.5 --skip-install --skip-git-init --pm npm
# Add to its package.json: "react-native-clear-signing": "file:../bindings/react-native"
# Add to its ios/Podfile: pod 'uniffi-bindgen-react-native', :path => '../node_modules/uniffi-bindgen-react-native'
cd ClearSigningExample
npm install
cd ios && bundle exec pod install && cd ..
npx react-native run-ios
```
