# React Native Integration

This guide covers the React Native SDK surface exposed by the `react-native-clear-signing` npm package.

The TypeScript surface is generated by [`uniffi-bindgen-react-native`](https://github.com/jhugman/uniffi-bindgen-react-native) on top of the existing UniFFI surface in [crates/clear-signing/src/uniffi_compat/mod.rs](../crates/clear-signing/src/uniffi_compat/mod.rs). Unlike Swift / Kotlin, there is no handwritten client wrapper — apps call the generated `clearSigning*` functions directly.

## What You Integrate

Import from the npm package:

```ts
import {
  clearSigningFormatCalldata,
  clearSigningFormatTypedData,
  clearSigningResolveDescriptor,
  clearSigningResolveDescriptorsForTx,
  clearSigningResolveDescriptorsForTypedData,
  type DataProviderFfi,
  type TransactionInput,
  type TokenMetaFfi,
  type FormatOutcome,
  FormatOutcome_Tags,
  type FormatFailure,
  FormatFailure_Tags,
  type DescriptorResolutionOutcome,
  DescriptorResolutionOutcome_Tags,
  type DisplayModel,
  type DisplayEntry,
  DisplayEntry_Tags,
  type DisplayItem,
  type FormatDiagnostic,
  type FallbackReason,
} from 'react-native-clear-signing';
```

App-facing entrypoints (all async):
- `clearSigningFormatCalldata`
- `clearSigningFormatTypedData`
- `clearSigningResolveDescriptor`
- `clearSigningResolveDescriptorsForTx`
- `clearSigningResolveDescriptorsForTypedData`

App-facing types:
- `DataProviderFfi` (interface — wallet-owned callback surface)
- `TransactionInput`, `TokenMetaFfi`
- `FormatOutcome` + `FormatOutcome_Tags`
- `FormatFailure` + `FormatFailure_Tags`
- `DescriptorResolutionOutcome` + `DescriptorResolutionOutcome_Tags`
- `DisplayModel`, `DisplayEntry` + `DisplayEntry_Tags`, `DisplayItem`
- `FormatDiagnostic`, `FallbackReason`

Tagged unions (`FormatOutcome`, `FormatFailure`, `DescriptorResolutionOutcome`, `DisplayEntry`) are discriminated by `tag` — match against the corresponding `_Tags` enum and read variant data from `inner`.

## Install The SDK

Requirements:
- React Native 0.81.5 or newer
- New Architecture enabled (`RCT_NEW_ARCH_ENABLED=1`)
- Hermes (default in RN 0.81)
- iOS 13.4+; Android `minSdk` 21, ABIs `arm64-v8a` / `armeabi-v7a` / `x86_64` (no `x86`); NDK + CMake required at app build time for the bundled Rust `.a` archives

### Published Consumption

```sh
npm install react-native-clear-signing uniffi-bindgen-react-native
```

`uniffi-bindgen-react-native` is the JSI runtime peer dep — install the version npm resolves; this release expects exactly `uniffi-bindgen-react-native@0.31.0-2` because the generated TS / C++ is runtime-version-coupled.

iOS: add to your app's `ios/Podfile`, inside the target block:

```ruby
pod 'uniffi-bindgen-react-native', :path => '../node_modules/uniffi-bindgen-react-native'
```

Then:

```sh
cd ios && bundle exec pod install
```

The `pod install` step downloads the prebuilt XCFramework from the matching GitHub Release and verifies its SHA-256 — no Rust toolchain needed on the consumer side.

Android: nothing extra. Autolinking picks up the package; rebuild with `npx react-native run-android`. The npm tarball ships stripped `libclear_signing.a` archives per supported ABI under `android/src/main/jniLibs/`; your app's gradle / CMake links them with the JSI bridge during the first build (adds ~2–3 min cold; subsequent builds incremental).

### Local Repo Development

The canonical local-dev consumer lives at [bindings/react-native/example/](../bindings/react-native/example) — a bare RN 0.81.5 app mirroring the iOS Wallet Debug tab. It consumes the package via `"react-native-clear-signing": "file:.."`.

Regenerate the package after editing the Rust crate:

```sh
./scripts/build-rn-ios.sh
./scripts/build-rn-android.sh
```

The iOS wrapper runs `ubrn build ios --sim-only --and-generate` and re-applies the podspec patches via `scripts/prepare-rn-podspec.sh`. The local XCFramework lands at `bindings/react-native/ClearSigning.xcframework/` (gitignored).

The Android wrapper runs `ubrn build android --and-generate`, then re-applies the 16 KB page-size CMake patch via `scripts/prepare-rn-android-cmake.sh`. The Rust static archives land under `bindings/react-native/android/src/main/jniLibs/` (gitignored).

## Integration Flow

Typical app flow:
1. Implement `DataProviderFfi`.
2. Pre-resolve descriptors asynchronously with `clearSigningResolveDescriptorsForTx(...)` (or the typed-data variant).
3. Pre-load any token / proxy / ENS / NFT metadata into in-memory maps your provider can read synchronously.
4. Call `clearSigningFormatCalldata(...)` (or `clearSigningFormatTypedData(...)`) and branch on `FormatOutcome.tag`.

`DataProviderFfi` callbacks are **synchronous across the FFI boundary** — they cannot return Promises. The wallet must therefore pre-fetch any data the formatter will need before calling format, so the callbacks can read from already-populated state. Step 2 (async resolve) is where you do the network work; step 4 (sync callbacks during format) reads from caches only.

The resolver performs descriptor resolution for the transaction or typed data, including nested calldata descriptors when needed. Proxy detection is delegated to your `DataProviderFfi.getImplementationAddress(...)`. Missing token / name / NFT metadata stays best-effort and surfaces as diagnostics, not hard failures.

Wallet policy should branch on `FormatDiagnostic.code`, not parse `FormatDiagnostic.message`.

## Implement DataProviderFfi

`DataProviderFfi` is the wallet-owned callback surface used for token metadata, naming, NFT metadata, block timestamps, and proxy resolution. Every method is synchronous.

Skeleton only:

This sketch is intentionally illustrative and omits concrete return statements.

```ts
import type {DataProviderFfi, TokenMetaFfi} from 'react-native-clear-signing';

const cacheKey = (chainId: bigint, address: string) =>
  `${chainId.toString()}:${address.toLowerCase()}`;

class WalletMetadataProvider implements DataProviderFfi {
  constructor(
    private readonly tokens: Map<string, TokenMetaFfi> = new Map(),
    private readonly proxies: Map<string, string> = new Map(),
    private readonly ensNames: Map<string, string> = new Map(),
  ) {}

  resolveToken(chainId: bigint, address: string): TokenMetaFfi | undefined {
    // Return token symbol, decimals, and name for this contract address.
    // Read from pre-populated caches — this is invoked synchronously.
  }

  resolveEnsName(address: string, chainId: bigint, types: string[] | undefined): string | undefined {
    // Return an ENS or other remote name for this address when available.
    // Return undefined when the wallet cannot resolve a name.
  }

  resolveLocalName(address: string, chainId: bigint, types: string[] | undefined): string | undefined {
    // Return a wallet-local contact or account label for this address.
    // Return undefined when no local label exists.
  }

  resolveNftCollectionName(collectionAddress: string, chainId: bigint): string | undefined {
    // Return the NFT collection name for this contract address.
    // Return undefined when unknown.
  }

  resolveBlockTimestamp(chainId: bigint, blockNumber: bigint): bigint | undefined {
    // Return the block timestamp for date-format rendering that depends on block numbers.
    // Return undefined when the wallet cannot look it up.
  }

  getImplementationAddress(chainId: bigint, address: string): string | undefined {
    // Return the proxy implementation address when this contract is a supported proxy.
    // Return undefined for non-proxies or when proxy detection is unavailable.
  }
}
```

Callback contract:
- `resolveToken`: used for token amount formatting and symbol / decimals / name display.
- `resolveEnsName`: used for remote address naming, such as ENS-style labels.
- `resolveLocalName`: used for wallet-local labels, address book entries, or "My Wallet"-style naming.
- `resolveNftCollectionName`: used when the descriptor wants a collection label for an NFT contract.
- `resolveBlockTimestamp`: used when descriptor rendering needs a block number converted to time.
- `getImplementationAddress`: used for proxy-aware descriptor resolution when `tx.to` does not directly match a descriptor.

A working seed-map implementation lives at [bindings/react-native/example/src/DemoDataProvider.ts](../bindings/react-native/example/src/DemoDataProvider.ts).

## Use The API Surface

### Resolve, Pre-load, Format Calldata

```ts
import {
  clearSigningResolveDescriptorsForTx,
  clearSigningFormatCalldata,
  DescriptorResolutionOutcome_Tags,
  FormatOutcome_Tags,
  type TransactionInput,
} from 'react-native-clear-signing';

const tx: TransactionInput = {
  chainId: 1n,
  to: '0xdAC17F958D2ee523a2206206994597C13D831ec7',
  calldataHex: '0xa9059cbb000000000000000000000000...',
  valueHex: undefined,
  fromAddress: '0x1234...',
};

// 1. Async resolve — hits the GitHub registry; falls back to proxy lookup via
//    DataProviderFfi.getImplementationAddress when needed. Pre-seed any
//    proxy implementations the wallet already knows.
const resolveProvider = new WalletMetadataProvider(new Map(), knownProxies);
const resolution = await clearSigningResolveDescriptorsForTx(tx, resolveProvider);
if (resolution.tag === DescriptorResolutionOutcome_Tags.NotFound) {
  return null;
}
const descriptors = resolution.inner[0];

// 2. Pre-load token metadata the formatter is likely to query. Wallets
//    typically inspect the calldata or descriptor here and fetch via their
//    own async paths (RPC, indexer, local cache).
const tokens = new Map<string, TokenMetaFfi>([
  [`${tx.chainId}:${tx.to.toLowerCase()}`, {symbol: 'USDT', decimals: 6, name: 'Tether USD'}],
]);

// 3. Format with a fresh provider holding the populated maps. The
//    DataProviderFfi callbacks fire synchronously during this call.
const formatProvider = new WalletMetadataProvider(tokens, knownProxies);
const outcome = await clearSigningFormatCalldata(descriptors, tx, formatProvider);

switch (outcome.tag) {
  case FormatOutcome_Tags.ClearSigned: {
    const {model, diagnostics} = outcome.inner;
    renderTrusted(model, diagnostics);
    break;
  }
  case FormatOutcome_Tags.Fallback: {
    const {model, reason, diagnostics} = outcome.inner;
    renderGeneric(model, reason, diagnostics);
    break;
  }
}
```

### Format Typed Data

```ts
import {
  clearSigningResolveDescriptorsForTypedData,
  clearSigningFormatTypedData,
} from 'react-native-clear-signing';

const resolution = await clearSigningResolveDescriptorsForTypedData(typedDataJson, provider);
if (resolution.tag === DescriptorResolutionOutcome_Tags.NotFound) {
  return null;
}
const outcome = await clearSigningFormatTypedData(resolution.inner[0], typedDataJson, provider);
```

### Resolve A Single Descriptor

```ts
import {clearSigningResolveDescriptor} from 'react-native-clear-signing';

const outcome = await clearSigningResolveDescriptor(1n, '0xdAC17F958D2ee523a2206206994597C13D831ec7');
if (outcome.tag === DescriptorResolutionOutcome_Tags.Found) {
  const descriptorJsonArray = outcome.inner[0]; // Array<string>
}
```

Use this when your app wants a direct lookup against the registry without a transaction context.

### Cancellation

Every async entrypoint accepts an optional `{ signal: AbortSignal }` second-argument-style options bag at the end:

```ts
const controller = new AbortController();
const outcome = await clearSigningFormatCalldata(
  descriptors,
  tx,
  provider,
  {signal: controller.signal},
);
// later, from another React effect cleanup or user action:
controller.abort();
```

## Returned Types

### FormatOutcome

`FormatOutcome` is the primary SDK result.

Variants (discriminated by `tag`):
- `FormatOutcome_Tags.ClearSigned` — `inner: { model: DisplayModel, diagnostics: FormatDiagnostic[] }`
- `FormatOutcome_Tags.Fallback` — `inner: { model: DisplayModel, reason: FallbackReason, diagnostics: FormatDiagnostic[] }`

Recommended wallet policy:
- `ClearSigned`: trusted clear-signing UI
- `Fallback`: generic / degraded UI with a visible fallback reason
- thrown `FormatFailure`: fail closed

Concrete fallback cases:
- descriptor not found for the contract
- known contract but no selector / encodeType format matched
- typed data missing `domain.chainId` or `domain.verifyingContract`
- nested calldata could not be clear-signed

### DisplayModel

`DisplayModel` is the render payload inside `FormatOutcome`.

```ts
type DisplayModel = {
  intent: string;
  interpolatedIntent?: string;
  entries: DisplayEntry[];
  owner?: string;
  contractName?: string;
};
```

### DisplayEntry

`DisplayEntry` is a tagged union for rendering rows.

Variants:
- `DisplayEntry_Tags.Item` — `inner: [DisplayItem]` (tuple of one)
- `DisplayEntry_Tags.Group` — `inner: { label: string, iteration: GroupIteration, items: DisplayItem[] }`
- `DisplayEntry_Tags.Nested` — `inner: { label: string, intent: string, owner?: string, entries: DisplayEntry[] }` (recursive for nested calldata; `owner` is the inner descriptor's `metadata.owner`, omitted on raw/fallback frames)

`DisplayItem` is the leaf row: `{ label: string, value: string }`.

A recursive renderer reference lives at [bindings/react-native/example/src/components/EntryRow.tsx](../bindings/react-native/example/src/components/EntryRow.tsx).

### FormatDiagnostic

`FormatDiagnostic` replaces legacy warning strings.

```ts
type FormatDiagnostic = {
  code: string;
  severity: DiagnosticSeverity; // 'Info' | 'Warning'
  message: string;
};
```

Contract:
- `code` is machine-readable and intended for wallet policy and telemetry
- `message` is human-readable and may evolve independently

Example:

```ts
if (diagnostics.some(d => d.code === 'nested_descriptor_not_found')) {
  showGenericNestedCallBadge();
}
```

### DescriptorResolutionOutcome

Resolution misses are explicit, not empty arrays.

Variants:
- `DescriptorResolutionOutcome_Tags.Found` — `inner: [string[]]` (tuple of one descriptor JSON array, in dependency order)
- `DescriptorResolutionOutcome_Tags.NotFound`

### TokenMetaFfi

`TokenMetaFfi` is the token metadata record returned from `resolveToken(...)`.

```ts
type TokenMetaFfi = {
  symbol: string;
  decimals: number;
  name: string;
};
```

## Error Shape

Async entrypoints reject with `FormatFailure`. Catch and discriminate by `tag`.

Variants (each `inner: { message: string, retryable: boolean }`):
- `FormatFailure_Tags.InvalidInput`
- `FormatFailure_Tags.InvalidDescriptor`
- `FormatFailure_Tags.ResolutionFailed`
- `FormatFailure_Tags.Internal`

Example:

```ts
try {
  const outcome = await clearSigningFormatTypedData(descriptors, typedDataJson, provider);
  // handle outcome
} catch (e: any) {
  if (e?.tag === FormatFailure_Tags.ResolutionFailed) {
    showResolutionError(e.inner.message, e.inner.retryable);
  } else if (e?.tag === FormatFailure_Tags.InvalidInput) {
    showBlockingError(e.inner.message);
  } else {
    showBlockingError(e?.message ?? String(e));
  }
}
```

`retryable` is intended for wallet policy and retry UX.
